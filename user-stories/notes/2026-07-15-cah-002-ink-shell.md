# 2026-07-15 CAH-002 Ink application shell

## Purpose

Record the concrete runtime, rendering, test, and terminal-lifecycle evidence that completed CAH-002
without implying that the Python process boundary or agent behavior exists.

## Decisions

- Pin Node 22.22.1 in `.node-version`, declare `>=22.13.0 <23` and npm 9 or newer in
  `tui/package.json`, and enable `engine-strict` in `tui/.npmrc`.
- Lock Ink 7.1.0 and React 19.2.7 in the npm project. JavaScript dependency changes commit
  `tui/package-lock.json`; they do not change Python's `uv.lock`.
- Use `scripts/run-tui` as the repository launch command. The POSIX launcher detects missing Node or
  npm, rejects a Windows Node executable reached through WSL, and validates the full supported range
  before invoking npm or its TypeScript loader.
- Keep version validation pure in `tui/src/node-version.ts`. `tui/src/bootstrap.ts` validates the
  explicit version before dynamically importing the Ink-owning module, which makes the pre-render
  boundary deterministic in tests.
- Let Ink own the CAH-002 Ctrl+C exit with `exitOnCtrlC: true`, and await `waitUntilExit()` before the
  CLI settles. There is no active session to cancel in this unit.
- Render a visibly static task-input area. Task entry and submission remain deferred rather than
  creating a local behavior that would be mistaken for the CAH-005 protocol path.

## Implemented path

`scripts/run-tui` validates the executable and version, resolves the repository, and runs the TUI's
`start` script. `tui/src/cli.ts` then calls `bootstrapApplication(process.versions.node)` as defense
in depth. After `assertSupportedNodeVersion` accepts the runtime, the bootstrap dynamically imports
`tui/src/run-application.tsx`. That module renders `tui/src/app.tsx` and waits for Ink to exit.

The initial frame contains these stable user-visible regions:

- `Code Assist Harness` title;
- `Conversation` with `No messages yet.`;
- `Task input` with an explicit disconnected-shell message; and
- `Status: idle · runtime not connected · Ctrl+C to exit`.

The shell is TypeScript throughout its production modules. Exported bootstrap, runtime, rendering,
and component contracts use TSDoc. The npm scripts are:

- `start` for the WSL shell;
- `typecheck` for strict TypeScript validation;
- `lint` for ESLint;
- `test` for a terminating Vitest run; and
- `check` for type checking, linting, and tests in dependency order.

## Test evidence

Six test files contain 15 passing tests:

- `tui/test/app.test.tsx` verifies the title, empty conversation, task input, status, and semantic
  region order with `ink-testing-library`;
- `tui/test/bootstrap.test.ts` proves Ink loads only after a supported runtime and is not loaded for
  an unsupported one;
- `tui/test/launcher.test.ts` proves a missing Node executable produces WSL-specific setup guidance
  and a fake unsupported Node exits before its fake npm can run;
- `tui/test/node-version.test.ts` covers supported boundaries plus missing, malformed, older, and
  newer versions;
- `tui/test/run-application.test.tsx` proves Ctrl+C cleanup is enabled and `waitUntilExit` is awaited;
  and
- `tui/test/runtime-metadata.test.ts` keeps `.node-version`, the npm engine range, and bootstrap
  constants aligned.

The tests inject versions and rendering seams rather than replacing the developer's active Node
installation or requiring a physical terminal. They make no provider or network request.

## WSL temporary-directory discovery

The implementation environment inherited `TEMP` and `TMP` values that pointed to a missing Windows
directory. Tools using the platform temporary-directory lookup could therefore fail even though the
repository and Linux `/tmp` were valid. The `start` and `test` scripts now set `TMPDIR=/tmp` narrowly
for those processes. This preserves the rest of the environment and records the actual WSL failure
rather than introducing broad environment rewriting.
